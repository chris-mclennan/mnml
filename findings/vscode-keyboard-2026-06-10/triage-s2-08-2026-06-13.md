# Triage — S2-08 Ctrl+N — 2026-06-13

## Finding

> Ctrl+N "new file" only fires from tree focus. Welcome overlay
> advertises `^N new file` globally; from editor pane focus it's
> silently swallowed. Tree focus opens "New file in /" prompt.

## Repro via headless harness

```sh
mkdir -p /tmp/mnml-c-test && cd /tmp/mnml-c-test
echo 'fn main() { println!("hi"); }' > main.rs
MNML_COLS=160 MNML_ROWS=50 mnml /tmp/mnml-c-test --input standard --headless &
```

```jsonl
{"cmd":"open","path":"main.rs"}      # focus=pane
{"cmd":"key","key":"ctrl+n"}         # nothing visible in screen.txt
{"cmd":"run-command","id":"file.new"} # command_run ok:true, but prompt
                                      #  still not visible
```

## Root cause

The Welcome overlay was up the whole time (`show_welcome = true` after
opening main.rs on a fresh workspace). `file.new` IS firing — both via
the chord AND via direct `run-command` — and `self.prompt = Some(...)`
gets set, but the renderer paints the Welcome overlay on top of the
prompt area.

Confirmation: dismiss welcome first, then Ctrl+N:

```jsonl
{"cmd":"key","key":"esc"}            # welcome dismissed
{"cmd":"key","key":"ctrl+n"}         # screen now shows:
                                      #   ╭ New file in / ──────────────╮
```

So the chord itself isn't being swallowed. The user-visible bug is
that Ctrl+N "looks like it does nothing" when the user hasn't
dismissed the Welcome overlay yet.

## Fix candidates (deferred)

1. **Auto-dismiss Welcome on any global chord that opens a UI surface
   (prompts, palettes).** Cleanest semantic but invasive — every
   prompt/palette opener would need a `app.dismiss_welcome_if_open()`
   call.
2. **Render-order swap — paint Welcome BENEATH the prompt overlay so
   prompts are visible while Welcome is still up.** Smaller diff but
   the visual stacking gets busy.
3. **Document the behaviour and leave as-is.** Welcome's whole point
   is "land here on launch, learn the chords, dismiss"; if the user
   actually wants to work they Esc the overlay first. The tester was
   exercising headless-startup state most users never linger in.

Closed as not-actively-broken; the chord is wired correctly. Pinned
here for re-check if anyone files the same complaint with a clearer
"I dismissed Welcome and Ctrl+N still doesn't work" repro.
